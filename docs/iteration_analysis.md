## 二、反复修改的根本原因

### 原因 1：没有反馈闭环（最致命）

```
当前流程：
  发现问题 → 改代码 → 跑测试（单元测试）→ ✅ 通过 → 提交
                                                    ↓
                                              但从未验证：
                                              改完之后盈利是否提升？
```

每次修改都"通过了测试"，但测试只验证**代码正确性**（"这个函数返回了预期值"），不验证**策略有效性**（"这个改动让盈利提高了"）。没有人知道第 9 次迭代的系统是否比第 6 次更赚钱。

**类比：** 你反复调整瞄准镜的刻度，但从未在靶场试射 → 不知道准不准。

### 原因 2：修一个 bug 引入一个新问题

| 迭代 | 修了什么 | 引入了什么新问题 |
|------|---------|-----------------|
| #4 加出场规则 | 有止损了 | 老仓位止损基于 avg_cost，永远不触发 |
| #7 加 NEUTRAL 门槛 0.90 | 熊市保护 | 太严格，合格信号被误杀 |
| #8 加 SIGNAL_TARGET | 中段减仓 | 需要 target_price 字段，数据全缺 |
| #9 加执行成本 0.5% 取消 | 控制滑点 | 最强的突破反而被取消（逆向选择） |
| #9 加行业集中度 | 分散风险 | 分母算错，有现金时权重翻倍 |
| #10 加 4 层 BEAR | 精细化风控 | 发现 market_regime 从未被传递过 |

这不是巧合，是**系统复杂度已超出人工 code review 的能力边界**。每加一个过滤器，它和现有过滤器的交互组合就呈指数增长。

### 原因 3：阈值在两个极端之间振荡

```
取消阈值:  0.5% → 1.5%  （第一次太严，过滤了好信号；放松后又担心接受了差信号）
NEUTRAL 门槛: 0.90 → 0.88  （太高误杀；降低后担心放进了噪声）
突破量能:  1.2× → 1.5× → 2.0×  （太松太多假突破；太紧漏掉真突破）
dte 窗口:  5-7 → 6-8 → 4-6  （日历日 vs 交易日搞混了）
lookback:  350 → 400 天  （第一次不够算 52 周高位）
时间止损:  20 → 45 天  （太短出太快；太长占用资金）
```

阈值调整是**成本最低但效果最差**的改动。没有回测数据支撑的阈值调整，本质上是在黑暗中打靶。

### 原因 4：规则加了但前置条件不满足 → 名存实亡

| 规则 | 加入时间 | 实际生效时间 | 名存实亡的时长 |
|------|---------|-------------|---------------|
| BEAR 市场屏蔽 | 2026-02-21 | 2026-04-04 | **42 天**（market_regime 从未传递） |
| 52 周高位加分 | 2026-03-13 | 2026-03-22 | **9 天**（lookback 不够 252 天） |
| SIGNAL_TARGET 减仓 | 2026-03-15 | **至今未生效** | 已 27 天（open_positions 无 target_price） |
| 时间止损 | 2026-02-19 | **至今未生效** | 已 51 天（open_positions 无 entry_date） |
| 亏损冷却期 | 2026-03-22 | **至今未生效** | 已 20 天（无 trades.json 文件） |

**这些"从未生效"的规则给人一种虚假的安全感** — 你以为风控在保护你，其实它从来没工作过。

### 原因 5：结构性缺陷 — LLM 作为决策者的不可控性

整个系统最终输出是一段 LLM prompt，让 GPT/Claude 做决策。但：
- LLM 可以忽略 prompt 中的规则（即使温度 = 0.1）
- Prompt 已经超过 1000 行，规则互相矛盾时 LLM 无法正确裁决
- 每次修阈值都要同步修 prompt → 经常漏改 → 代码和 prompt 不一致（已发生 4 次）

## 三、为什么一直在不停地改？

**本质答案：你在用"工程修补"来解决"策略验证"的问题。**

```
正确的循环（回测驱动）：
  假设 → 回测 → 量化结果 → 数据决定改什么 → 改 → 回测验证 → 提交

实际的循环（review 驱动）：
  读代码 → 觉得有问题 → 改 → 单元测试通过 → 提交 → 下一次读代码又觉得有问题
```

在没有历史数据回测的情况下，**每一次 code review 都会发现新的"问题"**，因为：
1. 任何阈值都可以被质疑（"为什么是 0.88 而不是 0.85？"）
2. 任何过滤器都可以被加强或放松
3. 任何边界条件都可以被更"精确"地处理

这创造了一个**无限循环** — 永远有东西可以改，但改了不知道是否更好。

## 四、建议：如何打破循环

### 1. 先建回测 → 再改策略（最高优先级）

用 `backtester.py`（已存在但未集成到日常流程中）对每次修改做 A/B 测试：
- 修改前跑一次回测，记录 Sharpe、最大回撤、胜率
- 修改后再跑一次
- **只提交让指标变好的改动**

### 2. 冻结阈值，聚焦数据基础设施

当前最大的问题不是"阈值不对"，而是：
- `open_positions.json` 缺少 `entry_date`、`target_price`、`cash_usd`
- 没有 `trades.json`（已关闭交易记录）
- 没有每日 PnL 曲线

**没有这些数据，任何策略优化都是盲目的。**

### 3. 减少过滤层级

当前信号要通过 7+ 层过滤才能到达 LLM：
```
策略条件 → ATR 门槛 → 量能门槛 → RS 门槛 → dte 门槛 → 置信度门槛 
→ R:R 门槛 → 行业集中度 → 组合热度 → TQS 门槛 → 取消阈值
```
每层 70% 通过率 → 0.7^11 ≈ 2% 总存活率。**过度过滤和不过滤一样有害。**

### 4. 将硬规则从 LLM prompt 移入代码

目前代码和 prompt 存在大量重复规则（dte 门槛、取消阈值、行业集中度等）。
**应该让代码做所有可量化的决策，LLM 只做"这条新闻是好是坏"的主观判断。**
这样不需要担心代码和 prompt 不同步的问题。

### 5. 停止每周改一次

设定一个季度节奏：
- 第 1 个月：收集数据（跑 pipeline，记录信号和结果）
- 第 2 个月：回测分析（哪些规则有效，哪些无效）
- 第 3 个月：批量调整 + 验证

**最贵的代价不是用错了阈值，而是永远不知道阈值对不对。**

## 五、迭代 15+（2026-04-17 → 现在）：系统已收敛，转入数据积累与归因

**状态**：2026-04-17 基线显示 8/8 criteria pass、`CONVERGED`、EV score = 0.4954、Sharpe 3.04、DD 4.59%。按 §五 原则，不再做策略微调，优先修**测量口径**与**归因可审计性**。

### exp-20260417-004（accepted）— LLM 决策回放基础设施
把 `known_biases.llm_gate_unreplayed` 从 bool(True) 升级为结构化 dict（`enabled` / `coverage_fraction` / `dates_covered` / `dates_missing_n`），新增 `result.llm_attribution` 桶与 `--replay-llm` flag。off-mode 数值与基线一致，on-mode 覆盖率仅 3.23%（5/124 天），数据量不足以做 alpha 归因，但机制已就位。

### exp-20260418-001（pending_validation）— 新闻档案覆盖率可审计化
沿用 exp-004 模式：把 `known_biases.news_veto_unreplayed` 从 bool(True) 升级为结构化 dict（`archive_replay_enabled` / `archive_coverage_fraction` / `archive_dates_covered` / `archive_dates_missing_n`）。每个 sim 日检查 `data/clean_trade_news_YYYYMMDD.json` 是否存在，纯测量，不改决策逻辑。数值指标由构造决定与基线 bit-identical。预测覆盖率约 10%（13/124 交易日），仍低于可做归因的阈值；但把"有多少档案可用"明确落盘后，下一轮可以在 news_replay.py 中基于 `archive_dates_covered` 做 T1 新闻规则回放并与策略结果做归因。

### 累计观察

1. **LLM 不是问题，覆盖率才是**：我们有 6 个 `llm_prompt_resp` 文件、20 个 `clean_trade_news` 文件，但 124 个交易日。不论策略改不改，覆盖率不上去，任何策略归因都带系统性偏差。这是下一季度的头号数据积累目标。
2. **已收敛 ≠ 可以继续微调**：策略层已满足全部 criterion；进一步动信号引擎 / 过滤器 / LLM prompt，在归因机制就位之前都属于"拿歪尺子量"（§6.1）。
3. **每一次"bool → 结构化 dict"的小动作都给未来留下杠杆**：exp-004 和 exp-20260418-001 都只是把一个 `True` 换成一个有字段的 dict，不改行为；但它们让下一次迭代可以写回放代码、跑归因、做 A/B 对比，而不必再返工补基础设施。
