# CLAUDE.md

一、工作顺序（必须严格遵守）
阶段1：建立事实基线（禁止修改代码）
阅读代码仓库，先识别并明确：
主策略入口
主回测入口
主配置文件
数据流与时间戳定义
信号生成链路
仓位 sizing / 风控 / 执行模拟链路
绩效评估模块
回测与实盘是否共用同一逻辑
将模块分级：
P0：直接影响交易决策 / PnL / 风险
P1：影响评估有效性
P2：工程结构
输出审计报告（在任何修改前完成）
二、审计报告要求（每个问题必须包含）
所在位置
问题类型：
（正确性 / 可交易性 / 风险控制 / 收益质量 / 工程）
为什么影响真实收益 / Sharpe / 回撤 / 成本 / 换手率
证据级别：
A：代码级直接证据
B：实验级证据
C：高风险推断（需验证）
修复优先级（P0/P1/P2）
最小修复方案
需要补充的验证实验
三、优先级规则（必须遵守）

按以下顺序处理：

回测失真问题（最高优先级）
数据泄漏 / 未来函数
时间对齐错误
指标或PnL计算错误
实盘不可执行问题
手续费 / 滑点 / 流动性缺失
成交/延迟假设不真实
风险失控问题
仓位 sizing 不合理
杠杆/回撤/尾部风险未约束
收益质量问题
过度交易 / 高换手
收益集中 / 参数不稳定
工程问题（最低优先级）
四、修改规则（仅针对高优先级问题）
每个修改必须先说明：
改动目的
影响范围
预期改善指标
为什么是最小必要改动
禁止：
为提高回测结果调参
无证据大改策略
允许（用于建立证据链）：
消融实验
延迟信号测试（+1 bar / tick）
成本敏感性测试
参数扰动测试
OOS / walk-forward 验证
五、修改后验证（必须全部完成）

不得只报告“测试通过”，必须提供：

修改前后回测对比（同口径）
成本前 vs 成本后收益
Sharpe / 回撤 / 收益回撤比
换手率 / 交易次数 / 持仓分布
延迟 / 滑点敏感性
参数扰动稳定性
OOS 或 walk-forward 表现
必须做收益归因：

说明收益变化来自：

alpha 提升
成本下降
风险收缩
或暴露增加
六、重点审计项（必须检查）
数据泄漏 / 未来函数
时间对齐问题
成本建模缺失
sizing / 风控缺陷
回测与实盘逻辑不一致
收益是否依赖极少数交易或时段
若包含 LLM：

必须检查：

是否诱导过度交易
是否缺乏 no-trade / abstain 机制
是否绕过风险约束
是否输出高置信度但无 edge
七、失败处理原则

如果修改后结果未改善：

禁止继续尝试新策略
必须先解释失败原因
更新证据链后再进入下一轮
八、严格禁止
过拟合历史数据
忽略手续费、滑点、流动性
无证据修改策略
用代码重构掩盖策略无效
用单次回测改善替代鲁棒性验证
