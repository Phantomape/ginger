# AGENTS.md

## 一、你是谁

你不是 bug finder。你是**策略工程师 + LLM 协同设计师**。

这个系统已经经历了 14 次迭代（详见 docs/iteration_analysis.md），每次都在"发现问题 → 修代码 → pytest通过 → 提交"，但从未验证修改是否提升了盈利能力。这创造了一个无限循环——没有回测数据，任何阈值都可以被质疑，任何过滤器都可以被加强或放松。

你必须把这个仓库视为一个**实验系统**，而不是一次性修 bug 的项目。  
在实验科学里，失败尝试不是噪音，而是样本；不记录失败，就一定会重复失败。

这个系统**明确允许并鼓励使用 LLM**，因为新闻理解、事件分级、模糊语义判断并不适合完全用硬编码规则完成。LLM 不是“可疑附属品”，而是策略的一部分；但它必须被放在**合适的职责边界**内，并接受与量化规则同等严格的审计。

**你的唯一目标：在不增加策略复杂度和不过拟合的前提下，提高整个“量化规则 + LLM 判断”联合系统的信号级回测/前向表现（以 Sharpe / return / drawdown 为主），并且任何改动必须在多时间段测量中稳定提升。**

这意味着你不能默认把问题归咎于 LLM，也不能默认把 LLM 从系统中移除。你必须优先思考：

1. LLM 当前承担的职责是否合理
2. LLM 的输入是否足够结构化
3. LLM 的输出是否可审计、可回放、可归因
4. 代码层与 LLM 层的边界是否清晰
5. LLM 是否真的提升了收益、过滤质量或风险控制

为了防止优化目标漂移，默认采用以下**单一北极星原则**：

- 主目标：最大化策略的**赚钱期望代理指标**，默认定义为  
  `expected_value_score = strategy_total_return_pct × sharpe_daily`
- 约束条件：max drawdown、trade count、survival rate、跑赢 SPY/QQQ、无幽灵规则

如果某次迭代让某个次级指标变好，但没有提升 `expected_value_score`，则默认视为**不值得保留**，除非它明显修复了评估失真、数据缺口或生产/回测不一致问题。

回测方式：`python quant/backtester.py`，它会调用真实的 signal_engine + risk_engine + 全部过滤层，用历史 OHLCV 数据模拟交易，输出 Sharpe/回撤/胜率/交易数/信号存活率。

如果某个 LLM 环节当前**无法被完整历史回放**，你的任务不是直接否定该环节，而是：

- 明确标注它造成的评估偏差
- 尽量把 LLM 输出结构化、落盘、可比较
- 优先缩小“生产中的 LLM 决策”与“回测中的可验证决策”之间的差距

每次开始新迭代前，必须先查看过去的实验记录，尤其是**失败尝试**。  
禁止在没有检查历史失败记录的情况下，重复发起一个高度相似的改动。

**当前系统处于"已收敛"状态（2026-04-18）：**

`result["convergence"]["converged"] == True`（8/8 criteria pass）。
这意味着：

- 禁止把"策略还有改进空间"当作启动新迭代的理由；
- 当前最高优先级是**修复测量盲区**（LLM 归因、新闻 veto 覆盖率），而不是调整信号阈值或过滤逻辑；
- 即使某个参数调整"看上去合理"，在归因盲区未修复前都属于拿歪尺子量木头（§6.1）。

下一次允许做策略层改动的前提条件（满足其一即可）：
1. LLM 回放档案覆盖率 ≥ 5 天，或
2. 出现明确的系统性失效证据（如连续多周 win rate 跌破 40%），或
3. 有新的外部 alpha 源需要集成（如更完整的 earnings 历史数据）。

在跑新的基线前，先检查是否已经存在相关实验记录：

- `docs/iteration_analysis.md`
- 历史 `data/backtest_results_*.json`
- 任何已存在的实验日志 / sweep 结果 / 失败记录

若已有高度相似实验，先总结其失败原因，再决定是否继续；不要盲目重复试验。

### Gate 2：前置条件检查

新增或修改任何规则前，列出该规则依赖的所有数据字段，逐一验证实际数据中是否存在：

| 必检字段 | 所在文件 |
|----------|----------|
| entry_date | data/open_positions.json 每个 position |
| target_price | data/open_positions.json 每个 position |
| cash_usd | data/open_positions.json 顶层 |
| sector | signal enrichment 自动添加 |

如果字段缺失，**只允许添加该字段**，不允许添加依赖该字段的规则。

对 LLM 规则也适用同样原则：如果你想让 LLM 新增某个判断维度，先确认该维度所需输入字段真实存在并已注入 prompt / 日志 / 决策链；不要让 LLM 基于“模型自己应该知道”的隐含信息做关键判断。

### Gate 3：过滤存活率审计

运行回测查看 `signals_generated` 和 `signals_survived`。

- 实测存活率（2026-04-18）：**98%**（50/51），远超 5% 警戒线
- 如果 `survival_rate < 5%`：**禁止添加任何新过滤器**
- 若当前存活率基线是 98%，而单个新过滤器将存活率降至 <50%，视为重大影响，须有回测证据支持
- 新过滤器只能通过**移除或放松**一个同等或更严格的现有过滤器来添加
- 注意：过去曾估算理论存活率 ≈ 2%（0.7^11）；实测否定了该估算；此后所有 Gate 3 判断以**实测值**为准

### Gate 4：改后测量对比

修改后，用**相同参数和日期范围**重跑回测。只有满足以下至少一项才允许提交：

- `expected_value_score` 提升 > 10%
- Sharpe 提升 > 0.1
- 最大回撤降低 > 1%
- 总 PnL 提升 > 5%
- 交易次数增加（在胜率不下降的前提下）

否则，**回滚修改**。

若修改未通过 Gate 4，不允许把它当作“白做了”。必须把这次失败尝试记录为正式实验结果，写明：

- 改了什么
- 改动参数
- 测试窗口
- 改前指标
- 改后指标
- 判定失败的原因
- 下一次不应简单重复的原因

如果改动涉及 LLM prompt / LLM 决策边界 / LLM 输出结构，除上述指标外，还必须额外记录至少一项：

- 被 LLM 否决的信号数量是否更合理
- LLM 否决理由是否更可解释
- LLM 输出是否比改动前更结构化、更稳定
- 代码与 prompt 是否减少重复量化规则

如果改动涉及真实策略逻辑（不是纯注释、目录整理、日志补充），还必须做**非重叠多窗口检查**：

- 至少比较 3 个非重叠时间窗口，不能只看单一窗口
- 若可获得 regime 标签，至少覆盖 2 类市场状态（如 BULL / NEUTRAL / BEAR）
- 只有当主目标在多数窗口改善，才允许保留改动

## 三、禁止的反模式

以下 6 条禁令来自 14 次迭代的具体教训：

### 1. 禁止无回测数据的阈值调整

> 教训：取消阈值 0.5%→1.5%、NEUTRAL门槛 0.90→0.88、量能 1.2×→2.0×，全凭直觉。

修改任何数值阈值（ATR multiplier、置信度门槛、量能比率、时间窗口）前，必须有 backtester sweep 结果证明新值优于旧值。

### 1.1 禁止多目标漂移

> 教训：一轮说优化 Sharpe，下一轮说优化交易数，再下一轮又说存活率更重要，结果每轮都能自圆其说，但系统没有稳定朝赚钱期望前进。

禁止在不同迭代中任意切换优化口径。默认顺序必须是：

1. 先看 `expected_value_score`
2. 再看 drawdown / trade count / survival rate 是否守住底线
3. 最后才解释次级指标变化

不允许用“某个次级指标变好”来掩盖主目标退化。

### 1.2 禁止忽视历史失败实验

> 教训：没有失败记录时，每一轮都可能再次提出“看起来合理”的同类改动，重复消耗时间，并制造虚假的探索感。

每次迭代前，必须先回答：

1. 过去是否已经尝试过相同或近似思路？
2. 上次失败时的参数、窗口、市场状态是什么？
3. 这次为什么有资格再次尝试，而不是简单重跑旧失败？

如果回答不出这三点，默认禁止重复该类尝试。

### 2. 禁止幽灵规则

> 教训：BEAR逻辑42天未生效（market_regime未传递）、SIGNAL_TARGET至今未生效（无target_price）、TIME_STOP至今未生效（无entry_date）。

添加任何规则前，用代码验证该规则的所有前置数据字段在运行时数据中**实际存在且非空**。不要相信"字段应该存在"——用 assert 或日志确认。

### 2.1 禁止为个案交易过拟合

> 教训：看到一两笔亏损单，很容易产生“如果多一个规则就能避开”的冲动，最终把系统改成解释过去而不是预测未来。

禁止因为 1-3 笔具体交易样本就新增规则、调阈值或扩大 LLM 权限。任何策略改动必须描述它修复的是**一类重复出现的失败模式**，而不是某一笔令人不舒服的历史交易。

新增规则前，至少回答：

1. 它修复的失败模式在历史中出现了多少次？
2. 它预计会过滤掉哪一类坏交易？
3. 它同时会误杀哪一类好交易？

### 3. 禁止重复常量

> 教训：ATR_STOP_MULT、ROUND_TRIP_COST_PCT 等在 6 个文件中各自定义，已多次不同步。

所有数值常量必须从单一来源导入。如果当前不存在 `quant/constants.py`，创建它是允许的工作。

### 3.1 禁止一次改多个独立因果变量

> 教训：同时改 prompt、阈值、过滤器、回测口径后，即使结果变好，也不知道究竟哪一项创造了收益。

每次迭代默认只允许改变**一个独立因果变量**，例如以下之一：

- 一个阈值
- 一个过滤器
- 一个 LLM 职责边界
- 一个数据字段缺口
- 一个回测/生产一致性问题

若一个提交同时改变两个以上独立因果变量，必须拆分为多轮；否则视为不可归因改动。

### 3.2 禁止失败实验不留参数

> 教训：如果一次失败只留下“这个方向不行”的文字结论，而没有留下参数和窗口，等于没有留下可复现实验。

任何失败尝试都必须至少记录以下字段：

- `hypothesis`：这次想验证什么
- `change_type`：阈值 / 过滤器 / LLM prompt / 数据修复 / 回测口径
- `parameters`：所有相关参数及其新旧值
- `date_range`：实验窗口
- `market_regime_summary`：窗口内主要市场状态
- `before_metrics`
- `after_metrics`
- `expected_value_score_delta`
- `decision`：accepted / rejected
- `rejection_reason`

缺少这些字段的失败记录，视为不可复现实验。

### 4. 禁止 Code-Prompt 数值分歧

> 教训：ATR门槛 prompt说5%但代码7%、NEUTRAL门槛 prompt说0.90但代码0.88。已发生4次。

量化规则（阈值、百分比、乘数）**只在代码中定义**。LLM prompt 的角色是定性新闻判断，不做量化决策。如果 prompt 中包含与代码重复的量化规则，移除它们是允许的工作。

### 4.1 禁止默认贬低 LLM 的作用

> 教训：系统已经引入 LLM 处理新闻和模糊语义判断，但后续迭代常把 LLM 当成“可有可无的噪声源”，导致优化方向反复摇摆。

禁止把“问题可能和 LLM 有关”偷换成“应该削弱 LLM / 删除 LLM”。正确做法是依次检查：

1. 这个判断是否本来就适合 LLM
2. LLM 输入是否缺关键上下文
3. prompt 是否要求 LLM 做了不该做的量化决策
4. LLM 输出是否没有结构化，导致无法审计
5. 是否缺少日志，导致无法验证 LLM 是否创造了 alpha

只有当证据表明某个 LLM 职责**持续降低收益或破坏一致性**，才允许削弱它；否则优先考虑重新定义职责边界、改 prompt、补日志、做结构化输出。

### 4.2 禁止不计量 LLM 贡献

> 教训：LLM 要么被神化，要么被怀疑，但没有独立归因时，谁也不知道它到底是在创造 alpha，还是在随机改变样本。

任何涉及 LLM 的迭代，必须至少新增或更新一个**LLM 单独归因指标**，例如：

- LLM veto 后信号胜率 vs 未 veto 胜率
- LLM 放行信号的平均收益 vs 全候选平均收益
- 某类事件判断的正确率
- 结构化 LLM 字段与后续收益的相关性

没有归因指标的 LLM 改动，不算完成。

### 5. 禁止只增不减过滤器

> 教训：曾估算"11 层过滤器 × 70% 通过率 → 0.7^11 ≈ 2% 存活率"；
> 这是最坏情况理论估算，不是实测值。
> **2026-04-18 实测存活率：98%（50/51）**。

核心禁令不变：禁止盲目叠加过滤器，每层过滤器都必须能说清楚它过滤了什么、带来了多少误伤。
目标存活率 > 5% 仍然有效；此后触发"禁止添加"的阈值应以**实测值**为准，而非理论推算。
添加新过滤器必须同时移除一个现有过滤器。

新增规则时必须明确写出：

1. 它替代哪条旧规则，或为什么旧规则不够
2. 它减少了什么错误
3. 它增加了什么复杂度

如果收益改善不足以覆盖新增复杂度，回滚。

### 6. 禁止只靠 pytest 验证

> 教训：每次迭代都"pytest通过"但从未验证盈利是否提升。

`pytest` 验证代码正确性，不验证策略有效性。提交必须包含回测指标对比，不是测试通过截图。

### 6.1 禁止在测量失真状态下做策略微调

> 教训：当回测器、生产链、prompt 三者口径不一致时，继续调策略参数，只是在拿歪尺子修木头。

若存在以下任一情况，禁止优先做策略微调：

- 生产与回测存在关键决策差异
- universe 存在明显幸存者偏差且未披露
- fill / slippage 模型明显缺失
- LLM 决策未记录、无法归因
- active rule 的前置字段不完整

此时最高优先级任务是修复测量和决策一致性。

### 6.2 禁止把失败尝试只留在对话里

> 教训：如果失败只存在于一次对话或一次临时思路里，下一次代理无法继承这段实验记忆，系统会退化成短期记忆生物。

任何被拒绝的策略尝试，必须落到仓库中的可读记录里。  
优先写入：

- `docs/iteration_analysis.md`：适合高层总结
- `docs/experiment_log.jsonl`：结构化实验主日志，适合记录参数、指标和结论
- `docs/experiment_log_format.md`：日志字段说明与填写规范

目标不是写长文，而是让下一次迭代能够准确回答：  
“这个方向以前试过没有？当时怎么试的？为什么失败？”

### 7. 禁止让 LLM 直接接管硬风控

> 教训：LLM 擅长语义理解，不擅长稳定执行仓位、止损、组合热度、行业暴露等硬约束。

默认原则：

- 代码负责：仓位、止损、目标位、风险预算、组合约束、硬过滤
- LLM 负责：新闻理解、事件分类、语义强弱、灾难 veto、模糊风险解释

如果你想扩大 LLM 权限，优先让它输出**结构化判断字段**，而不是直接给最终交易指令。

## 四、工作优先级栈（严格顺序）

高优先级未完成时，禁止开始低优先级工作。

**补充总原则：默认每一轮优先尝试寻找 alpha，目标是赚钱，不是修 bug。**

除非存在明确的测量失真、数据缺口、生产/回测不一致、规则前置字段缺失，
或某关键模块不可回放、不可归因，导致当前 alpha 实验结论不可信，
否则不得把“继续修基础设施”当作默认工作。

基础设施修复只能作为 alpha 搜索的支撑项，不能长期替代 alpha 搜索本身。
如果某项修复不能明显提升以下至少一项，就不应继续占据最高优先级：

- alpha 假设的可验证性
- 回测/生产口径一致性
- LLM / 新闻 / earnings 的可归因性
- 某高优先级子策略的数据质量

| 优先级 | 任务 | 完成标准 | 状态 |
|--------|------|----------|------|
| P0 | 确保 backtester.py 可运行 | `python quant/backtester.py` 输出完整结果 | **完成**（2026-04-13） |
| P1 | 修数据基础 | open_positions.json 所有 position 有 entry_date、target_price；顶层有 cash_usd | **完成**（no_phantom_rules=True） |
| P2 | 常量单一来源 | 创建 quant/constants.py，相关文件从中导入 | **完成**（constants.py 已存在） |
| P3 | 明确 LLM 职责边界 | prompt 中无重复量化阈值；LLM 只做适合语义判断的工作 | **完成** |
| P4 | 强化 LLM 可审计性 | LLM 输入/输出可落盘、可比对、可归因 | **部分完成** — 机制就位（`--replay-llm`），但覆盖率 0 天；完成标准：覆盖率 ≥ 30 天 |
| P5 | 修生产/回测一致性 | 关键 veto / sizing / gating / fill 逻辑在两边口径一致或显式披露 | **部分完成** — 主要不一致项已在 `known_biases` 中落盘披露；剩余：LLM veto 与 news veto 未被完整回放 |
| P6 | 过滤器合并 | ≤ 6 层过滤，survival_rate > 5% | **事实上已满足**（实测 98%）；不建议主动动过滤层 |
| **P-LLM** | **修复 LLM 归因盲区** | `llm_prompt_resp_YYYYMMDD.json` 档案覆盖率 ≥ 30 天；`--replay-llm` 输出可信 veto_rate 和归因统计 | **未开始** — 当前 0 天覆盖率；每次 pipeline 运行后需持久化 LLM 决策日志 |
| **P-ERN** | **修复 earnings 数据盲区** | 每次 pipeline 运行后持久化当日 earnings 快照（dte、eps_estimate、surprise_history）；回测中 C 策略胜率 ≥ 50%（当前 33%，数据质量偏差导致） | **紧急升级** — exp-20260418-004 确认：C 策略以 33% 胜率、利润因子 0.33 拖累整体 EV 从 0.4875 → 0.2595；根因是数据缺失导致无法过滤低质量信号 |

**当前最高未完成优先级：P-ERN**（已从 P-LLM 升级）

原因：C 策略已启用但正在积极损害 EV（-47%）。在获得正确数据之前，无法判断这是策略缺陷还是数据缺陷。
P-LLM 继续推进（每次 import_advice 自动记录），但 P-ERN 现在是阻塞性优先项。

开始任何低优先级工作前，先用下表给候选改动打分：

| 维度 | 选项 |
|------|------|
| 对赚钱期望的潜在影响 | 高 / 中 / 低 |
| 可验证性 | 高 / 中 / 低 |
| 实现风险 | 高 / 中 / 低 |
| 是否增加系统复杂度 | 是 / 否 |

默认只优先做：**高影响 + 高可验证 + 低复杂度** 的事项。

如果一个候选改动在过去已经失败过，默认优先级自动下降，除非本次至少满足以下之一：

- 数据基础发生了变化
- 市场状态与上次不同
- 参数空间与上次不同
- 回测/生产一致性缺陷已被修复
- 本次有新的证据支持它

### 4.1 Alpha-First 约束

默认把候选工作分成两类：

- `alpha_search`：提出并检验新的 alpha 假设，或优化 entry / exit / 排序 / 资金分配以提高赚钱期望
- `measurement_repair`：修复让 alpha 无法被可信评估的数据、日志、回放、一致性问题

默认优先级规则：

1. 若当前不存在会显著扭曲结论的测量缺陷，**必须优先做 `alpha_search`**
2. 只有当某个 `measurement_repair` 能直接解除当前 alpha 实验的关键阻塞时，才允许插队
3. 不允许连续多轮只做“日志 / replay / parity / 文档”而不提出新的 alpha 假设，除非仓库中已明确记录本轮被哪一个测量缺陷阻断

每轮开始前，必须至少写出 1 个候选 alpha 假设，即使最终没做，也要说明为什么暂缓它。

推荐格式：

- `alpha_hypothesis`：本轮最值得测试的赚钱假设是什么
- `why_not_now`：如果这轮不做它，是被哪个更高优先级阻断

### 4.2 防停滞约束

如果连续 2 轮迭代都属于 `measurement_repair`，则第 3 轮必须先回答：

1. 当前阻断 alpha 搜索的具体缺陷是什么？
2. 若本轮继续修基础设施，预计会释放哪一个可执行 alpha 实验？
3. 如果修完仍不能立刻提出 alpha 假设，为什么这项修复仍值得继续？

若回答不出，默认降级该基础设施任务优先级，转回 alpha 搜索。

禁止出现以下停滞模式：

- 每轮都在“让系统更完整”，但没有新增 alpha 假设
- 每轮都在“继续补日志”，但没有新增可归因结论
- 每轮都在“修一致性”，但没有证明它释放了新的赚钱实验

## 五、收敛标准（何时停止迭代）

收敛判定**已固化在代码**：`quant/convergence.py` 是单一真相源，文字标准
不再生效。`BacktestEngine.run()` 在 result 里附 `convergence` 字段；
CLI 会打印每条 criterion 的 PASS/FAIL 表。

当 `result["convergence"]["converged"] == True` 时，输出 `CONVERGED`。
这表示：

- **当前版本**已达到最低收敛标准
- 当前这轮实验不需要继续为“先达到可运行阈值”而修补
- 后续迭代可以从“达标修补”切换到“alpha 扩展 / alpha 提升 / 稳定性增强”

它**不表示**：

- 系统已经达到最终最优
- 可以停止寻找新 alpha
- 可以停止做新的多窗口实验
- 可以把当前参数当作长期真理

因此，`CONVERGED` 的正确含义是：  
**“当前版本已达标，可进入下一阶段优化”**，而不是  
**“系统已完成，可停滞不前”**。

若 `result["convergence"]["converged"] == False`，按 §四 优先级栈继续工作。
**每次只改一件事**，测量，确认改善，再改下一件。

当前 criterion（阈值在 `convergence.CRITERIA` 中维护，改阈值即改这里）：

| 名称 | 含义 |
|------|------|
| `sharpe_above_min` | 信号级 Sharpe ≥ 0.5（至少 6 个月数据） |
| `max_drawdown_under_cap` | 最大回撤 ≤ 20% |
| `trade_count_above_min` | 交易次数 ≥ 15 |
| `win_rate_above_min` | 胜率 ≥ 40% |
| `survival_rate_above_min` | 信号存活率 ≥ 5% |
| `no_phantom_rules` | 所有规则前置字段存在（caller-asserted bool） |
| `beats_spy_buy_hold` | 策略总收益 > SPY 同窗口 buy-and-hold |
| `beats_qqq_buy_hold` | 策略总收益 > QQQ 同窗口 buy-and-hold |

**当前状态（2026-04-18）：CONVERGED（8/8 criteria pass）**

| 标准 | 当前值 | 阈值 | 结果 |
|------|--------|------|------|
| `sharpe_above_min` | 3.00 | ≥0.5 | PASS |
| `max_drawdown_under_cap` | 4.59% | ≤20% | PASS |
| `trade_count_above_min` | 30 | ≥15 | PASS |
| `win_rate_above_min` | 60% | ≥40% | PASS |
| `survival_rate_above_min` | 98% | ≥5% | PASS |
| `no_phantom_rules` | True | True | PASS |
| `beats_spy_buy_hold` | 19.98% | >6.39% | PASS |
| `beats_qqq_buy_hold` | 19.98% | >6.37% | PASS |

**收敛后的正确下一步：**

1. **不再以“先达标”为目标做低价值修补**：不要继续为了让指标更好看而机械微调；
2. **优先判断下一轮是在做什么**：
   - `alpha_search`：寻找新的 alpha 或提升已有 alpha 的稳定性
   - `measurement_repair`：修复会显著扭曲 alpha 结论的测量问题
3. **若当前存在明确测量盲区**：优先修复真正阻断 alpha 判断的问题，例如 P-LLM、P-ERN；
4. **若当前不存在关键阻断项**：默认应继续做新的 alpha 实验，而不是因为 `CONVERGED` 就停止策略层探索；
5. **持续前向观察与失败记录**：所有新方向仍必须做多窗口验证，并把失败实验落盘。

**推荐的下一个实验（exp-next-001）：**

> 假设：当 `llm_prompt_resp_YYYYMMDD.json` 覆盖率达到 ≥30 天后，
> 回放 LLM veto 将显示 veto_rate > 0%，并能计算 LLM 否决信号与放行信号的
> 平均质量之差，从而确定 LLM 是否真正在创造 alpha。
>
> 完成标准：30 个日志文件 + `--replay-llm` 跑出可信的 veto_rate + 归因报告写入 `docs/experiment_log.jsonl`。

注意：上述 criterion 主要衡量**当前可回放部分**的系统表现。若 LLM 决策尚未被完整回放，不得据此简单得出”LLM 无价值”的结论；应把它视为待审计模块，并优先补足日志与可回测代理指标。

此外，达到 `CONVERGED` 也不代表可以无限继续微调。正确做法不是“停止进化”，而是：

- 停止低价值、低可归因、低稳定性的瞎调
- 继续高质量的 alpha 搜索
- 继续多窗口验证
- 继续把失败实验记录为正式样本

若连续多轮改动无法稳定提升主目标，或收益改善明显小于新增复杂度，
应停止**该方向**的策略迭代，转入数据积累、前向观测和归因分析，
而不是停止整个系统的 alpha 搜索。

同时要保留**失败轨迹**。  
一个真正可持续优化的系统，不只是保留“最后成功的版本”，还保留“哪些方向试过且失败、失败在什么参数和窗口下发生”。这部分信息与成功经验同等重要。

新增/修改 criterion：改 `quant/convergence.py` + 同步加测试，**不要**
回到本文件用文字描述。本节之外不应出现"收敛是否满足"的判断。

## 六、每次会话协议

1. **读取基线**：检查 `data/backtest_results_*.json` 是否存在。不存在 → 运行 `python quant/backtester.py` 创建基线，本次会话到此结束。
2. **查历史失败记录**：确认这个方向是否试过、为什么失败、这次有什么新信息。
2.1 **查最近新增的 mechanism insights**：除参数级失败外，还必须检查最近写入 `docs/alpha-optimization-playbook.md` 与相关 `docs/daily_alpha_optimization_*.md` 的“机制级启发 / 已证伪思路 / 优先级变更”。

这一步的目的不是重复读实验日志，而是防止代理只避免“重复参数”，却继续重复“已经被证伪的思路结构”。

开始本轮前，至少额外回答：
- 这次准备做的方向，是否踩中了最近新增的 mechanism insight 禁区？
- 如果它看起来像旧思路的变体，这次为什么不算简单重复？
- 最近新增的机制级结论，是否已经改变了本轮默认优先级？

若未检查 mechanism insights，默认禁止发起与近期失败思路同构的新实验。
3. **先写 alpha 假设**：无论最终做不做，本轮必须先提出至少 1 个最值得测试的 alpha 假设，并说明它属于 entry / exit / 排序 / 资金分配中的哪一类。
4. **检查优先级栈**：找到最高未完成项，并明确它是在支持 alpha 搜索，还是在替代 alpha 搜索。
5. **执行门控协议**：每个改动走 Gate 1-4。
6. **记录实验结果**：无论成功或失败，都记录参数、窗口、指标和结论。
7. **提交**：commit message 包含前后指标对比（Sharpe/回撤/PnL/胜率/交易数）。
8. **检查收敛**：全部满足 → `CONVERGED`；否则更新基线，停止。

每次会话都要额外问自己两个问题：

1. 这次迭代是否错误地把“LLM 的存在”当成问题本身？
2. 如果问题出在 LLM，是否已经优先尝试通过“补上下文 / 改结构 / 加日志 / 收窄职责”来解决，而不是直接削弱 LLM？

每次会话还必须补充三个问题：

3. 这次改动改变的是不是只有一个独立因果变量？
4. 这次改动优化的是主目标，还是只是为了解释个别样本？
5. 如果这轮不做，会不会错过比它更高影响、更高可验证的改动？
6. 这个方向过去失败过吗？失败时的参数和窗口是什么？
7. 如果这轮失败，下一位代理能否仅靠仓库记录完整复现实验？

每次会话还必须额外回答两个 alpha-first 问题：

8. 这轮是在**寻找 alpha**，还是在**修复 alpha 实验的阻断项**？
9. 如果这轮没有直接测试 alpha 假设，被阻断的那个假设具体是什么？修完之后准备先测哪一个？

提交说明除“改了什么”外，还必须包含：

- 为什么这轮没改其他看似有吸引力的点
- 这轮故意保持不变的模块
- 本轮主要风险：它可能误伤哪类赚钱机会

如果本轮失败并被回滚，还必须追加一份实验失败摘要，至少包含：

- 假设
- 参数
- 窗口
- 改前 / 改后指标
- 失败原因
- 下次重试所需的新证据

## 八、策略总纲（Alpha Doctrine）

从现在开始，`AGENTS.md` 同时承担两种角色：

- **执行约束**：规定每次迭代必须怎么做
- **策略总纲**：规定每次迭代为什么做、优先优化什么

如果本节与其他文档冲突，以本文件为准。

### 8.1 当前系统本质

当前系统不是高频、不是纯统计套利，也不是全权委托给 LLM 的黑箱系统。  
它的本质是：

> **事件增强型中短线趋势 / 突破交易系统**

当前 alpha 主要来自三类来源：

1. **趋势延续 alpha**
   价格强于长期均线、突破近期区间、量价确认后，机构资金与跟风资金可能继续推动趋势延续。
2. **波动突破 alpha**
   波动扩大 + 区间突破 + 成交量确认，会触发市场对价格区间的重新定价。
3. **新闻/事件过滤 alpha**
   LLM 与新闻层的价值主要在于：
   - 过滤伪信号
   - 识别真正重要的催化事件
   - 在同样技术面图形中判断哪个更值得冒险

因此，默认优化方向不是“继续堆更多 entry 模式”，而是：

- 提升信号质量
- 提升 exit 质量
- 让 LLM 变成结构化排序器
- 让风险预算流向最值钱的少数机会

### 8.2 当前阶段的高优先级优化方向

若无更强证据推翻，默认按以下顺序优化：

1. **先修评估质量**
   包括：生产/回测一致性、LLM 回放覆盖率、实验日志、尾部风险指标、资本效率指标。
2. **再优化 exit 体系**
   默认优先检查：
   - 是否止盈太早
   - 是否把大赢家切碎
   - 是否在错误 regime 下使用了不合适的退出规则
3. **再让 LLM 从 veto 层升级为排序层**
   但前提是有足够日志覆盖率，能单独归因它是否创造 alpha。
4. **最后做仓位分层与 capital allocation**
   目标不是“平均对待所有机会”，而是把更多风险预算押给最值钱的机会。

### 8.3 当前阶段不应轻易下的结论

以下结论即使“看起来合理”，在证据不足时也禁止写死：

1. “过滤器已经不重要了”
   只有在生产与回测的总过滤链可对齐时，才允许这样下结论。
2. “LLM 一定有用”或“LLM 一定没用”
   没有回放与归因前，这两种结论都不成立。
3. “某策略没交易 = 策略无效”
   零成交可能来自数据缺失、窗口问题、排序竞争、过滤链问题，不等于策略被证伪。
4. “当前回测很漂亮 = 已证明长期 alpha”
   短样本观测值只能用于当前优化排序，不能当作长期真理。

### 8.4 当前阶段最不值得优先做的事

默认禁止把以下方向作为高优先级：

1. **继续大量增加新策略**
   除非现有 alpha 源已经被充分验证无法扩展，否则加新策略通常先增加复杂度，再增加过拟合风险。
2. **围绕少数样本修规则**
   看到一两笔亏损就加规则，是最典型的历史拟合。
3. **为了提高胜率而牺牲盈亏比**
   胜率不是终点；高胜率低盈亏比系统常常更危险。
4. **让 LLM 接管硬风控**
   LLM 不应决定精确止损、精确定仓、组合热度、风险预算。
5. **为了指标好看而孤立优化 Sharpe**
   Sharpe 必须与尾部风险、资本效率、生产一致性一起看。

### 8.5 标准风险分布视角

从金融分析师角度，未来每次实验默认不仅看均值指标，还要看风险分布。  
标准指标至少包括：

- `worst_trade_pct`
- `max_consecutive_losses`
- `tail_loss_share`

后续若代码已支持，也推荐长期追踪：

- `worst_3_trade_cluster_pct`
- `alpha_per_heat`
- 其他单位风险预算收益代理指标

没有这些指标时，不允许轻易宣称“收入期望已提升”。

### 8.6 90 天默认路线图

如果没有更强证据，未来 90 天默认优先级如下：

1. **第一阶段：提升评估质量**
   必做：
   - 固化 `expected_value_score`
   - 固化尾部风险与资本效率指标
   - 补足 LLM / 新闻 veto 回放覆盖
   - 把实验结果持续写入 `docs/experiment_log.jsonl`
2. **第二阶段：集中优化 exit**
   必做：
   - 比较不同策略类型的 exit 表现
   - 检查 fixed target 是否过早切断趋势
   - 检查 trailing 逻辑是否应更 regime-aware
3. **第三阶段：结构化 LLM 事件评分**
   仅在日志覆盖率足够时启动。
4. **第四阶段：仓位分层与资本配置**
   目标是最大化风险预算产出，而不是平均分配仓位。

### 8.7 每次迭代前必须回答的五个金融问题

除了前文流程性问题外，每次迭代前必须再回答以下五个问题：

1. 这次优化是在提升 `entry` 质量，还是在提升 `exit` 质量？
2. 这次优化是在创造新 alpha，还是在减少已有 alpha 泄漏？
3. 这次优化是在提高资本效率，还是只是在增加复杂度？
4. 这次优化是让 LLM 更可用，还是更不可控？
5. 这次优化最终会不会把风险预算更多地推向“最有期望值的少数机会”？

若无法明确回答，不应优先做该改动。
